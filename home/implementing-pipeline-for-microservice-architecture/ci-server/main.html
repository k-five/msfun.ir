ci-server

We talked about task runners, right? A CI Server in the simplest form is a task runner. A task runner, runs some tasks for us based on our need. A CI Server is a huge task runner with much capabilities that can managers many tasks.

For example Travis-CI is a CI Server.

What does it do?. Very simple. We sign-up there and register a task to be run.

Task for what?. For a project (= repository) we have.

Where is this repository?. Well, on our github account.

How does it work with our repository?. After registering a project for Travis-CI, it wants us to let it to have access to our Github repository. Then if we give this access to Travis-CI, on that specific project (= repository) it sets up GitHub-WebHooks. Then based on these events have been registered for GitHub-WebHook, GitHub will notify Travis-CI for change and Travis-CI based on the task we give to it, run some tasks for us. Then based on the result of tasks, Travis-CI modify a SVG file - we have it on our README.md - to show us if the tests / tasks have been failed or succeeded.

FAQ

Is it good to have a CI Server?

Yes. It help us to separate our build process from our source code and shared repository.

Do we need a CI Server for our own?

No, if we have a simple project, but for experimental purpose - fun - why not.

And if a project is big? There are many services out there, you can search for CI Servers / Services.
