ci-server

We talked about task runners, right? A CI Server in the simplest form of a task runner. A task runner, runs some tasks for us based on our needs. A CI Server is a huge task runner with much capabilities that can manage many tasks.

For example Travis-CI is a CI Server.

What does it do?. Very simple. We sign-up there and register a task to be run.

Task for what?. For a project (= repository) we have.

Where is this repository?. Well, on our github account.

How does it work with our repository?. After enabling / registering a project for Travis-CI, it wants us to let it to have access to our Github repository. Then if we give this access to Travis-CI, on that specific project (= repository) it sets up GitHub-WebHooks. Then based on some events that have been registered for GitHub-WebHook, GitHub will notify Travis-CI for changes and Travis-CI based on the task we give to it, runs the task for us. Then based on the result of tasks, Travis-CI modify an SVG file - we have it on our README.md - to show us if the tests / tasks have been failed or succeeded.

FAQ

Is it good to have a CI Server?

Yes. It helps us to separate our build process from our source code and shared repository.

Do we need a CI Server for our own?

No, if we have a simple project, but for experimental purpose - fun - why not.

And if a project is big? There are many services out there, you can search for CI Servers / Services.

At this point you should have a picture and the overall workflow of a simple pipeline. Which are

On our local machine

  • We modify our source code
  • some task runners will be run, if we have
  • some tests will be run, if we have
  • some build locally will be run, if we need
  • we continuously commit to our local repository
  • we push to our shared repository

On our shared repository (e.g. GitHub)

  • file / directory watchers on GitHub servers will detect change (after our push)
  • The services on our shared repository will notify (e.g. GitHub Web-Hook) our CI Server or other CI Server.
  • Our or other CI Servers synchronize themselves with the shared repository
  • file / directory watchers on CI servers will detect change (after synchronizing)
  • some files will be build, if we have
  • some tests will be run, if we have
  • with some CI Servers may our built code be sent to another server (e.g. Production Server)

On our production server

  • we will have production-ready code
  • the business will deliver it, if it decide to do so

Finally we are done. This was the end of our simple pipeline with microservices.
