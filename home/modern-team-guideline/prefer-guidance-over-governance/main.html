prefer-guidance-over-governance

Imagine a King that wants to control everything and everybody. We can name this behavior Centralized Governance. In this model one person tries to dictate everything to others

The inverse of this model is to have several people - e.g a Congress - so the decisions can be made by discussing and not dictating. We can name this behavior Decentralized Governance.

Monolithic versus Microservices

Now if we, our team, our company has a Large Code Base that controls every aspect of the application, business, or software we have - in the same manger like the king - then we have a Monolithic Architecture.

The inverse of this model can be, breaking down key functionalities of our large code base to small - almost - independent chucks of code. Doing so we can have Service Based Architecture which in a precise and modern team we can call it Microservice Architecture.

Does it make sense when we have a Congress, then just one person decides to dictate everything? No.

The same is true for Developing Software.

When we break our large code base to smaller services, and we give each service to a person, or a team to manage, then it does not make sense to force them to choose a specific Technology Stack.

Guidance and not Governance

This rule emphasizes on NOT to force a team to follow a Hero in the company whoever who might be.

  • CTO
  • PM
  • etc

Teams should be able to choose the technology that want to develop their services based on.

Not doing so means while physically we have a Microservice Architecture but culturally we have a Monolithic Architecture.

As a manager when we give our teams position, we should give them freedom to choose, too.

This is true for example for Implementation Stack, but not for Monitoring. Because in terms of monitoring, for the entire services we have, we should have consistency.

Therefore it is preferable to be guidance for our teams / developers and not governance. E.g forcing everybody / every team to use / choose MongoDB as their database is not preferable.
