incidental-coupled-team

This model of organizing teams is the old one in which we organize teams around / based on Job Function.

What is job function? Simple. The position the people take to do a specific job.

In this model we really do not have a team. What do you have then? A group of related positions.

And these people are organized around / based on their jobs. So we have front-end team, back-end team, etc.

Conway's Law

We have a very interesting law named Conways's Law.

organizations which design systems ... are constrained to produce designs which are copies of the communication structures of these organizations.

M. Conway

What does it mean?

The software and application we design is kind of a system, right? If we have bad organized organization and communication, then this reflect to our code and finally our application.

Usually we will end up having N-tiered Layered.

And not surprisingly having Monolithic Architecture.

Focus of the team

In this model focus of the team is not on value nor is it on products.

Where is the focus on then? I do not know :(.

Project

In this model someone (= could be CTO, PM, etc) gives a team or a person a task or a portion of code to be completed by them; and they will do it.

The developers in this model code and deliver the code to whoever gives them the task.

Who should care about the rest of the operation to the code?

  • deployment
  • testing
  • building if we have any kind of it
  • etc

Usually the person who get the code! The hero, the super man of the company.

So how about front-end developers or back-end, etc? Hmmm ... No one cares.

Problem

Okay what is the problem with this model?

The first one to me is wasting time to deliver code. Read this: You build it, you run it.

The second one is not having a Pic Picture in mind of the developers about the overall of the application or the software they coded.

The third can be not feel responsible for code after deliver it.

etc.

Solution

Very simple. Inverse Conway Maneuver
