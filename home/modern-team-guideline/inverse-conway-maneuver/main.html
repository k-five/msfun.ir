inverse-conway-maneuver

With the inverse of conways's maneuver, instead of focusing on developers job function, we focus and products.

Instead of saying we have project to finish we say we have product to deliver.

And instead of organizing developers based on their job title, e.g front-end, back-end, we organize them based on Bounded Context.

Bounded Context is one of keys in DDD (= Domain Driven Design).

Team

In this model teams are not based on a specific position, e.g all front-ends developers but on the requirements of a service or product we have.

We have a service that needs only one person? So that person will take the responsibility of that service. One other service needs a designer, a front-end, a back-end, an Oracle Database Specialist? Then these four become a team for that service.

Focus is on the product, deliver product and not code or deliver source code to someone else!

So the goal is to have an autonomous team that can take care of their source code till putting it on production.

Responsibility

In this model we give developers much more responsibility than the old model we discussed in other posts. That is way the CTO of Amazon says: You build it, your run it.

You start the source code, so you should take care of it till deliver it to production server.

Pipeline

Because of these model many interesting things come up.

E.g how a front-end developer can deliver code?

This is where Development Operation (= DevOps) comes to the play. And not just physically but also culturally.

Having a pipeline helps teams or organizations to know where they are going. Also which and how many steps they have before they could deliver their product / value to their customer.

And good pipeline is based on good automation. All the key steps from committing to the main / shared repository up to the production server should be automated.

Every team should be able to deploy every day or even any hour if they want - which usually is.
