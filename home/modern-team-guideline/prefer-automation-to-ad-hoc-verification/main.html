prefer-automation-to-ad-hoc-verification

One of the most interesting part of microservices architecture is to have a pipeline.

Because we have many operations to do, it does not make sense to do all operations manually. And we do not have a pipeline unless we have automation.

For example

  • tests automation
  • deployments automation
  • builds automation
  • automation in monitoring. e.g reporting failure in a service

why automation

The key thing about automation is not just that we do not have to do things manually, but mostly it is that we no longer have to worry about that part in our pipeline, our system, etc.

In fact we can forget about it and just focus on more important things.

example

For example for writing this blog I have set up a simple automated pipeline;

  • editing a file and commit the change on my local machine
  • git push to Github
  • GitHub will receive changes and notify the Node.js server which takes care of this blog
  • Then it checks if notification is from GitHub, if yes, it does git pull and then merge the changes to existing files.
  • if required the Node.js server will be restarted
  • If something goes wring I will get emails, text messaging, etc

These are happen automatically. I just write and push. And no longer think about the rest. So my focus is on writing and not:

  • uploading files
  • create files on the server
  • get new files from GitHub
  • merge files
  • ssh to server and check for failures
  • etc

So avoid checking your services using ssh.

For example for monitoring we can have a Flat TV in our IT room and in real-time watch the state of our services.

Not only we save time, but also are confident about status of our services.

add more stage

One other good thing about having a pipeline and automation system is that if right now we do not have a stage for testing in our pipeline, we can add one more easily comparing to do it manually with our existing stages.
